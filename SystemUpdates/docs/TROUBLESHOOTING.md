# Troubleshooting guide

Begin with these commands:

```bash
sudo systemctl status ollama.service --no-pager -l
sudo systemctl status open-webui.service --no-pager -l
sudo podman ps -a
ollama list
curl -v http://127.0.0.1:11434/api/tags
curl -v http://127.0.0.1:3000/
```

For the older generated service, replace `open-webui.service` with
`container-open-webui.service`.

## Open WebUI says it cannot reach `host.containers.internal:11434`

Typical message:

```text
Cannot connect to host host.containers.internal:11434
Connect call failed ('10.88.0.1', 11434)
```

Ollama is probably listening only on `127.0.0.1`.

Check:

```bash
sudo ss -lntp | grep 11434
```

Install or verify:

```text
/etc/systemd/system/ollama.service.d/override.conf
```

It must contain:

```ini
[Service]
Environment="OLLAMA_HOST=0.0.0.0:11434"
Environment="OLLAMA_NUM_PARALLEL=1"
Environment="OLLAMA_MAX_LOADED_MODELS=1"
```

Apply it:

```bash
sudo systemctl daemon-reload
sudo systemctl restart ollama.service
sudo ss -lntp | grep 11434
```

Test from inside the running container:

```bash
sudo podman exec open-webui \
  python -c "import urllib.request; print(urllib.request.urlopen('http://host.containers.internal:11434/api/tags').read().decode())"
```

Do not solve this by opening `11434/tcp` to the LAN.

## Open WebUI says `Failed to fetch models`

First verify Ollama itself:

```bash
curl -fsS http://127.0.0.1:11434/api/tags
ollama list
```

Then test the container-to-host path using the command above. Confirm the
Open WebUI connection is:

```text
http://host.containers.internal:11434
```

Restart both services:

```bash
sudo systemctl restart ollama.service
sudo systemctl restart open-webui.service
```

## The WebUI opens but the model list is empty

Check whether models are installed:

```bash
ollama list
```

Install one if needed:

```bash
ollama pull gemma3:4b
```

Restart Open WebUI and force-refresh the browser:

```bash
sudo systemctl restart open-webui.service
```

- macOS: `Command+Shift+R`
- Windows/Linux: `Ctrl+F5`

## Friendly model names are missing

Friendly names are not created by `ollama pull`. They are created with
`ollama create`.

List the templates:

```bash
sudo find /etc/ollama/modelfiles \
  -maxdepth 1 \
  -type f \
  -iname '*.modelfile' \
  -print
```

Validate their `FROM` lines:

```bash
sudo grep -Hi '^[[:space:]]*FROM[[:space:]]' \
  /etc/ollama/modelfiles/*.modelfile
```

Rebuild all:

```bash
for file in /etc/ollama/modelfiles/*.modelfile; do
  name="$(basename "$file" .modelfile)"
  ollama create "$name" -f "$file"
done
```

Use lowercase filenames. Version 2.2.1 of `updates.sh` discovers those files
and rebuilds them during each daily run.

## The update script tries to pull a friendly model from the registry

Verify the installed version:

```bash
sudo /usr/local/sbin/updates.sh --version
```

Expected:

```text
updates.sh 2.2.1
```

Reinstall the repository version when needed:

```bash
cd /opt/scripts
sudo git pull --ff-only
sudo install -o root -g root -m 0750 \
  SystemUpdates/updates.sh /usr/local/sbin/updates.sh
```

Run a preview:

```bash
sudo /usr/local/sbin/updates.sh --only-ai --dry-run --verbose
```

The preview should pull base names such as `qwen3:8b`, but should rebuild
rather than pull names such as `general-assistant:latest`.

## `Permission denied` after typing a `.service` path

A systemd unit is configuration, not an executable program. This fails:

```text
/etc/container-open-webui.service
```

Use `systemctl`:

```bash
sudo systemctl daemon-reload
sudo systemctl start container-open-webui.service
```

For Quadlet, place the source file at:

```text
/etc/containers/systemd/open-webui.container
```

and manage the generated service:

```bash
sudo systemctl daemon-reload
sudo systemctl start open-webui.service
```

## `Unit container-open-webui.service not found`

A file generated by `podman generate systemd --files` is written to the
current working directory. Systemd does not scan `/etc` itself for arbitrary
files; normal unit files must be under `/etc/systemd/system`.

Locate it:

```bash
sudo find / -name 'container-open-webui.service' 2>/dev/null
```

If retaining the legacy unit:

```bash
sudo mv /path/to/container-open-webui.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now container-open-webui.service
```

For a future rebuild, use the repository Quadlet instead.

## A second `mv` says the service file does not exist

It may already have been moved. Check the destination:

```bash
ls -l /etc/systemd/system/container-open-webui.service
```

Do not regenerate or overwrite a working service until its configuration and
persistent volume have been backed up.

## Podman says `no container with ID or name open-webui`

A generated `--new` service may remove its transient container when it stops.
The named volume can still exist.

Check:

```bash
sudo podman ps -a
sudo podman volume inspect open-webui
```

Starting the systemd service recreates the container:

```bash
sudo systemctl start container-open-webui.service
```

or:

```bash
sudo systemctl start open-webui.service
```

Never remove the `open-webui` volume while preserving data matters.

## Quadlet does not generate `open-webui.service`

Check the source:

```bash
sudo ls -l /etc/containers/systemd/open-webui.container
sudo systemctl daemon-reload
sudo systemctl cat open-webui.service
```

Ask the generator to show errors:

```bash
sudo /usr/lib/systemd/system-generators/podman-system-generator --dryrun
```

Also verify:

```bash
sudo systemd-analyze generators=true verify open-webui.service
podman info --format '{{.Host.CgroupsVersion}}'
```

Quadlet requires cgroup v2. An unsupported key or syntax error can prevent the
generated service from appearing.

## `open-webui` container name is already in use

Do not run the legacy unit and Quadlet simultaneously.

Inspect ownership:

```bash
sudo systemctl status container-open-webui.service --no-pager
sudo systemctl status open-webui.service --no-pager
sudo podman ps -a --filter name=open-webui
```

Choose one service. To migrate to Quadlet:

```bash
sudo systemctl disable --now container-open-webui.service
sudo podman rm -f open-webui 2>/dev/null || true
sudo systemctl daemon-reload
sudo systemctl start open-webui.service
```

The named volume is not deleted by `podman rm`.

## Port `3000` is unreachable from another computer

Confirm local health:

```bash
curl -I http://127.0.0.1:3000/
sudo ss -lntp | grep ':3000'
```

Check Fedora's active zone and rules:

```bash
sudo firewall-cmd --get-active-zones
sudo firewall-cmd --list-all
```

Ensure the client subnet is allowed to reach `3000/tcp`. Do not open
`11434/tcp`.

Confirm the client is using the server's current address:

```bash
hostname -I
```

## Open WebUI is active but the page is stale or broken after an update

Force-refresh the browser:

- macOS: `Command+Shift+R`
- Windows/Linux: `Ctrl+F5`

Then inspect logs:

```bash
sudo journalctl -u open-webui.service -n 200 --no-pager
```

A container downgrade may not work after a one-way database migration. Restore
the Open WebUI data volume from a pre-update backup when necessary.

## The daily timer did not run

```bash
sudo systemctl status local-daily-updates.timer --no-pager
systemctl list-timers local-daily-updates.timer
sudo journalctl -u local-daily-updates.timer --since yesterday --no-pager
sudo journalctl -u local-daily-updates.service --since yesterday --no-pager
```

Enable it:

```bash
sudo systemctl enable --now local-daily-updates.timer
```

Check the next scheduled time:

```bash
systemd-analyze calendar '*-*-* 03:15:00'
```

## The update script reports another run is active

The lock protects against overlapping package and model updates.

Check for a running process:

```bash
pgrep -af '/usr/local/sbin/updates.sh'
sudo systemctl status local-daily-updates.service --no-pager
```

Do not delete the lock file while a process is running. If no process exists,
a normal `flock` lock is released automatically; retry the script.

## Updates fill the SSD

Inspect usage:

```bash
df -h /
ollama list
sudo podman system df
```

Remove only models that are no longer referenced:

```bash
ollama rm MODEL:TAG
```

List Modelfile dependencies first:

```bash
sudo grep -Hi '^[[:space:]]*FROM[[:space:]]' \
  /etc/ollama/modelfiles/*.modelfile
```

Avoid `podman volume prune` because it can remove valuable application data.

## SELinux errors

Keep SELinux enforcing. The supplied configuration uses a named Podman volume,
which is preferable to an unlabeled arbitrary host directory.

Inspect recent denials:

```bash
sudo ausearch -m AVC,USER_AVC -ts recent
```

Verify the Open WebUI volume:

```bash
sudo podman volume inspect open-webui
```

Do not disable SELinux as a first troubleshooting step.

## Collect a diagnostic bundle

This command writes service and configuration metadata but deliberately does
not copy the WebUI secret or chat database:

```bash
OUT="/tmp/local-llm-diagnostics-$(date +%Y%m%d-%H%M%S)"
mkdir -p "$OUT"

sudo systemctl status ollama.service --no-pager -l >"$OUT/ollama-status.txt" 2>&1 || true
sudo systemctl status open-webui.service --no-pager -l >"$OUT/open-webui-status.txt" 2>&1 || true
sudo journalctl -u ollama.service -n 200 --no-pager >"$OUT/ollama-journal.txt" 2>&1
sudo journalctl -u open-webui.service -n 200 --no-pager >"$OUT/open-webui-journal.txt" 2>&1
ollama list >"$OUT/ollama-list.txt" 2>&1 || true
sudo podman ps -a >"$OUT/podman-ps.txt" 2>&1 || true
sudo ss -lntp >"$OUT/listeners.txt" 2>&1 || true
sudo firewall-cmd --list-all >"$OUT/firewall.txt" 2>&1 || true

tar -C /tmp -czf "${OUT}.tar.gz" "$(basename "$OUT")"
echo "${OUT}.tar.gz"
```

Review the archive before sharing it.
